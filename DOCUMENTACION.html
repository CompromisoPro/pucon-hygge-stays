1 Punto de partida — el prototipo GenSpark

El sitio fue generado originalmente por GenSpark AI, una herramienta que produce sitios web estáticos. El resultado era visualmente bueno pero completamente no funcional a nivel de backend:

  • Todas las llamadas a API apuntaban a tables/... (un servidor ficticio de GenSpark que no existe fuera de su plataforma)
  • No había base de datos real conectada
  • Las imágenes eran referencias a rutas locales (images/201/...) que no existían en el servidor
  • El panel de administración existía en HTML pero no podía guardar ni cargar nada
  • No había forma de subir fotos
⚠️
El sitio se veía bien en la demo de GenSpark porque ellos sirven una versión local. Al sacarlo de GenSpark, el sitio quedaba completamente vacío y sin funcionalidad.

2 Arquitectura elegida

Se decidió mantener el frontend estático (HTML/CSS/JS puro) y conectarlo a Supabase como backend. Esta decisión fue tomada por simplicidad, costo cero y compatibilidad total con Netlify.

🌐 Frontend

  • HTML + CSS + JavaScript puro
  • Sin frameworks ni build step
  • Diseño 100% preservado del original
  • Deploy en Netlify

🗄️ Backend

  • Supabase (PostgreSQL)
  • API REST automática (PostgREST)
  • Storage para imágenes
  • Row Level Security (RLS)

Flujo de datos

Visitante → index.html → cms-loader.js → Supabase API → muestra contenido
Admin    → admin/index.html → apiGet/Post/Patch/Delete → Supabase API → guarda cambios
Foto     → admin sube archivo → uploadToStorage() → bucket "fotos" → URL pública

3b. Storage — bucket de fotos

Se usa Supabase Storage para almacenar todas las imágenes del sitio. Hay que crear el bucket manualmente:

1
Ir a Supabase → Storage En el menú lateral del proyecto.
2
New bucket Nombre: fotos (exactamente así, minúsculas)
3
Marcar "Public bucket" Esto permite que las imágenes se vean en el sitio sin autenticación.
4
Save

3c. Políticas RLS de Storage

Aunque el bucket es público, Supabase requiere políticas explícitas para permitir la subida de archivos con el anon key. Correr en SQL Editor:

-- Permitir subida de archivos al bucket fotos
create policy "Permitir subida publica a fotos"
on storage.objects for insert
with check (bucket_id = 'fotos');

-- Permitir lectura publica
create policy "Permitir lectura publica de fotos"
on storage.objects for select
using (bucket_id = 'fotos');

-- Permitir actualizacion (para upsert)
create policy "Permitir actualizacion en fotos"
on storage.objects for update
using (bucket_id = 'fotos');

-- Permitir eliminacion
create policy "Permitir eliminacion en fotos"
on storage.objects for delete
using (bucket_id = 'fotos');
🚨
Sin estas políticas, la subida de fotos falla con el error: "new row violates row-level security policy". Si ves ese error, correr el SQL de arriba.

6 Subida de imágenes (Supabase Storage)

Todas las secciones del admin que tienen imágenes usan un componente de upload unificado que:

  • Tiene un botón "Subir foto" que abre el selector de archivo
  • Muestra una barra de progreso mientras sube
  • Muestra preview de la imagen subida
  • También acepta pegar una URL directamente
  • Tiene botón para limpiar/cambiar la imagen

URL generada

Cada imagen subida genera una URL pública permanente del tipo:

https://[proyecto].supabase.co/storage/v1/object/public/fotos/[carpeta]/[timestamp]-[random].[ext]

Esta URL se guarda en la tabla correspondiente de Supabase y es cargada automáticamente por el sitio público.

Notas de seguridad

  • El anon key es una clave pública. Es normal que esté en el frontend y en GitHub. No da acceso de admin a Supabase.
  • Lo que protege la base de datos son las políticas RLS (Row Level Security) configuradas en el SQL de creación de tablas.
  • La contraseña del admin del sitio (hygge2025) es simple y está en el código. Para producción seria se debería implementar autenticación con Supabase Auth.
  • Para rotar el anon key: Supabase → Settings → API → "Reset anon key".

11 SQL completo de referencia

Tabla galeria_fotos (correr además del SQL principal)

create table if not exists galeria_fotos (
  id          uuid primary key default gen_random_uuid(),
  imagen_url  text not null,
  titulo      text,
  categoria   text default 'general',
  orden       int default 1,
  activo      boolean default true,
  created_at  timestamptz default now()
);

alter table galeria_fotos enable row level security;

create policy "Public read galeria_fotos"
  on galeria_fotos for select using (true);

create policy "Admin insert galeria_fotos"
  on galeria_fotos for insert with check (true);

create policy "Admin update galeria_fotos"
  on galeria_fotos for update using (true);

create policy "Admin delete galeria_fotos"
  on galeria_fotos for delete using (true);
